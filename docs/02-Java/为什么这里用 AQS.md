# 为什么这里用 AQS

## 背景

AQS 是 JDK 并发包中锁、信号量、栅栏等同步器的基础框架。平台业务一般不需要频繁手写 AQS，但理解 AQS 有助于判断锁和线程阻塞行为。

## 适用场景

- 需要理解 `ReentrantLock` 的加锁和排队机制。
- 需要分析线程池、阻塞队列或限流组件中的线程等待问题。
- 需要实现极少数业务专用同步器。
- 需要排查高并发下的锁竞争和线程堆栈。

## 为什么关注 AQS

- 能理解独占锁和共享锁的底层差异。
- 能理解线程从运行态进入等待队列的过程。
- 能更准确地分析死锁、饥饿、吞吐下降等问题。
- 能帮助选择 `synchronized`、`ReentrantLock`、`Semaphore` 等工具。

## 平台中的可能应用

- 判题 Worker 并发控制。
- 容器创建限流。
- 用户提交频率保护。
- 排行榜刷新任务互斥。

## 注意事项

- 优先使用 JDK 已有同步工具。
- 只有在标准工具无法表达需求时才考虑自定义 AQS。
- 自定义同步器必须覆盖完整测试，包括中断、超时和异常场景。
