# 用户系统

## 模块定位

用户系统负责 Version 0.1 的账号注册、登录认证、当前用户身份识别和基础权限控制。

这个版本只支持普通用户，不做管理员后台、团队系统和复杂权限矩阵。

## Version 0.1 功能范围

1. 用户注册
2. 用户登录
3. 密码哈希存储
4. JWT 签发
5. JWT 校验
6. 获取当前登录用户
7. 禁止未登录用户访问受保护接口

## 暂不支持

1. 用户资料编辑
2. 用户头像
3. 找回密码
4. 邮箱验证码
5. 多角色权限
6. 管理员禁用用户
7. 团队创建与加入
8. 登录日志

## 核心实体

### 用户

用户是平台的基本参与者。

用户可以注册、登录、查看题目、提交 Flag、参与排行榜。

核心字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 用户 ID |
| `username` | 用户名 |
| `email` | 邮箱 |
| `password_hash` | 密码哈希 |
| `role` | 用户角色 |
| `status` | 用户状态 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

## 用户角色

Version 0.1 只需要两个角色。

| 角色 | 说明 |
| --- | --- |
| `USER` | 普通用户 |
| `ADMIN` | 管理员，暂时只预留，不做后台功能 |

## 用户状态

| 状态 | 说明 |
| --- | --- |
| `ACTIVE` | 正常用户，可以登录和提交 Flag |
| `DISABLED` | 禁用用户，不能登录和访问受保护接口 |

Version 0.1 可以先只创建 `ACTIVE` 用户，但数据库和代码里要预留状态字段。

## 注册规则

1. 用户名不能为空。
2. 用户名长度建议为 3 到 32 个字符。
3. 用户名只能包含字母、数字、下划线和短横线。
4. 邮箱不能为空。
5. 邮箱格式必须合法。
6. 密码不能为空。
7. 密码长度至少 8 位。
8. 用户名必须唯一。
9. 邮箱必须唯一。
10. 密码不能明文存储，必须保存哈希值。

## 登录规则

1. 用户可以使用用户名或邮箱登录。
2. 登录时必须校验密码哈希。
3. 用户不存在时，返回统一的账号或密码错误，避免泄露账号是否存在。
4. 密码错误时，返回统一的账号或密码错误。
5. 用户状态不是 `ACTIVE` 时，禁止登录。
6. 登录成功后签发 JWT。

## JWT 规则

JWT 用于后端识别当前登录用户。

请求受保护接口时，前端需要在请求头中携带：

```text
Authorization: Bearer <token>
```

JWT 中至少包含：

| 字段 | 说明 |
| --- | --- |
| `sub` | 用户 ID |
| `username` | 用户名 |
| `role` | 用户角色 |
| `iat` | 签发时间 |
| `exp` | 过期时间 |

Version 0.1 可以先设置 24 小时过期。

## 权限边界

### 公开接口

1. 注册
2. 登录

### 需要登录的接口

1. 查看题目列表
2. 查看题目详情
3. 提交 Flag
4. 查看排行榜

排行榜理论上可以公开，但 Version 0.1 为了统一认证流程，先要求登录。

```text

service

repository

```

## 关键异常

| 场景 | 建议错误码 | HTTP 状态 |
| --- | --- | --- |
| 用户名已存在 | `USERNAME_ALREADY_EXISTS` | 409 |
| 邮箱已存在 | `EMAIL_ALREADY_EXISTS` | 409 |
| 账号或密码错误 | `INVALID_CREDENTIALS` | 401 |
| 用户被禁用 | `USER_DISABLED` | 403 |
| Token 缺失 | `UNAUTHORIZED` | 401 |
| Token 无效或过期 | `INVALID_TOKEN` | 401 |

## 编码注意点

1. 密码必须使用 BCrypt 或同等级算法哈希。
2. 注册和登录接口不能返回 `password_hash`。
3. 用户名和邮箱唯一性要同时在业务层和数据库层保证。
4. JWT 密钥不能写死在代码里，应该从配置或环境变量读取。
5. 后端从 JWT 中解析用户 ID 后，仍然要查询数据库确认用户存在且状态正常。
